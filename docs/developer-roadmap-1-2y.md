# Service Router 开发者导向路线图（1~2年）

## 1. 文档目的

本路线图用于指导 `service-router` 在未来 1~2 年从“可用组件”演进为“开发者友好型路由产品”，核心关注开发效率（DX）、联调体验、排障效率与团队协作能力。

## 2. 产品定位（开发者导向）

- 面向对象：后端开发者、测试开发者、小中型服务团队
- 核心价值：让开发者以低心智负担完成本地联调、规则调试、集成测试与环境一致性验证
- 价值主张：更快跑起来、更快定位问题、更快协作上线

## 3. 路线策略

- DX-first：优先投入 CLI 易用性、可观测调试、测试友好能力
- Local-first：优先优化本地开发与联调链路，再扩展企业治理
- Progressive：能力按“开箱即用 -> 可诊断 -> 可协作 -> 可扩展”递进
- Measurable：每阶段绑定可度量指标，避免“功能完成但价值不显著”

### 4.1 远期规划（不设固定季度）

- **配置界面（GUI / Web）**：以可视化方式编辑与预览路由、注册中心等板块，降低 YAML 门槛；**不作为** DX 主线前置能力，与「YAML + CLI + GitOps」长期并存。落地形态可先偏向 **本地轻量工具**，团队级控制面属于可选增强。
- 与 Phase A/B 边界一致：不提前以「重型控制面」阻塞当前「开箱即用 + 可诊断」主线。

### 4.2 流量入口：**显式代理端口（保留）**，**叠加透明/中继入口（远期）**

为便讨论，沿用以下称谓（仅为文档用语，不与 backlog ID 混淆）：

- **模型 A（主线、保留）**：客户端（浏览器或其它进程）**显式**将 HTTP 发往路由器配置的监听地址（`server.host` / `server.port`），再由路由规则与注册中心解析转发。这是当前的实现与 CI/文档验收的**默认基准**。
- **模型 B（远期、可选叠加）**：在**不改动**模型 A 的前提下，通过 **操作系统端口转发（如 nftables/iptables REDIRECT、Windows portproxy）、用户态中继（如 socat）或远期可选的小型 companion**，将发往**特定本地端口**的流量**汇入**路由器监听端口，从而复用同一套路由与转发逻辑。  
  - **边界**：不承诺内核级捕获「所有进程、所有端口」的流量；明文 HTTP 场景优先；HTTPS 若要透明劫持需终结证书或 MITM，**复杂度高**，单独立项评估。  
  - **交付形态预期**：先有**文档化 playbook / 脚本**，再视需求讨论是否并入仓库或二进制。

## 5. 分阶段目标与边界

### Phase A（Y1 H1）：开箱即用 + 本地联调

- 目标：开发者 10 分钟内完成首次代理请求
- 重点：初始化脚手架、运行检查、基础联调路径
- 边界：不提前建设重型控制面与复杂组织治理

### Phase B（Y1 H2）：可诊断 + 可测试

- 目标：把“能用”提升为“可快速定位问题”
- 重点：命中解释、调试链路、mock 与 CI 集成
- 边界：避免过早引入高复杂度插件生态规则

### Phase C（Y2 H1）：可扩展 + 可协作

- 目标：支持团队级规范化与可复用能力沉淀
- 重点：插件 SDK、规则模板、版本对比与评审流
- 边界：控制扩展接口复杂度，保障升级兼容

### Phase D（Y2 H2）：生态化 + 一致性治理

- 目标：形成开发者生态与多环境一致性保障能力
- 重点：IDE 工具、模板市场、漂移检测、预演
- 边界：企业级控制面作为可选增强，不阻塞开发体验主线

## 6. 核心能力地图

### 6.1 基础体验层（must-have）

- `service-router init`：项目初始化与模板选择
- `service-router doctor`：依赖、端口、配置、连通性检查
- `service-router run --dev`：开发模式启动

### 6.2 调试诊断层（high-value）

- `route explain`：说明匹配了哪条规则，以及为什么匹配/未匹配
- request trace：请求在路由与上游的全链路日志
- 错误提示分级：配置错误、注册中心错误、上游错误可区分

### 6.3 测试集成层（developer workflow）

- mock registry：本地模拟服务实例注册发现
- 配置静态检查：`check-config`
- CI 冒烟：`smoke-proxy`

### 6.4 扩展生态层（long-term moat）

- 插件 SDK：请求前后处理、规则扩展点
- 模板市场：规则模板、场景模板
- IDE 支持：补全、校验、快速预览
- **配置界面（远期）**：见上文 **§4.1**；与 CLI/文档协作，而非替代配置即代码

## 7. 依赖与风险

- 兼容性风险：插件机制引入后，版本兼容策略必须前置设计
- 复杂度风险：调试能力与日志粒度增加可能带来性能开销
- 资源风险：若团队规模有限，需优先保证 L1/L2 能力落地
- 用户教育风险：新能力必须配套文档、样例与命令提示

## 8. KPI 与验收建议

### 北极星指标（North Star）

- TTV（Time to First Value）：首次成功代理请求所需时间
- DTR（Debug Time Reduction）：排障平均耗时下降比例
- SSR（Self-Serve Rate）：无需平台/中间件介入的自助解决率
- CCR（Config Change Reliability）：配置变更一次成功率

### 半年验收门槛（建议）

- Y1 H1：TTV < 10 分钟，核心流程文档覆盖率 100%
- Y1 H2：DTR 降低 >= 40%，CI 冒烟接入率 >= 60%
- Y2 H1：SSR >= 70%，插件能力试点团队 >= 30%
- Y2 H2：CCR >= 95%，环境漂移类故障下降 >= 50%

## 9. 版本节奏建议

- 双月小版本（功能增量）
- 半年中版本（阶段验收）
- 年度大版本（能力收敛与生态升级）

通过“短周期交付 + 阶段里程碑验收”，确保路线图既可执行又可纠偏。
