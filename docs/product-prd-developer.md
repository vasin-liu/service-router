# Service Router 产品需求文档（PRD，开发者导向）

## 1. 文档信息

- 产品名称：Service Router
- 文档版本：v1.0
- 文档类型：PRD（开发者导向）
- 对齐周期：未来 1~2 年

## 2. 背景与问题陈述

当前 `service-router` 已具备基础路由、注册中心解析、热更新和 HTTP 代理能力，但在开发者高频场景中仍存在问题：

- 上手路径不够短，首次成功成本偏高
- 联调与排障依赖经验，缺少“为什么命中/未命中”的解释能力
- 测试集成能力不完整，难以在 CI 中稳定复现问题
- 团队协作缺乏配置版本化与评审闭环

## 3. 目标与非目标

### 3.1 产品目标

- G1：显著降低首次接入与首次成功时间（TTV）
- G2：显著降低路由相关问题排障时间（DTR）
- G3：构建可复用、可协作、可扩展的开发者工具链
- G4：在 1~2 年内形成开发者生态能力（插件 + 模板 + IDE）

### 3.2 非目标（当前阶段不优先）

- 全量企业控制面（复杂审批系统、组织级权限体系）
- 大规模跨地域流量治理的重平台能力
- 与服务网格同级的全栈治理覆盖

## 4. 用户与核心场景

### 4.1 目标用户

- 个人开发者/小团队后端开发
- 测试开发与集成测试工程师
- 应用团队技术负责人（关注协作与规范）

### 4.2 核心场景

- S1：本地启动并快速接入一个新服务
- S2：联调中快速判断路由命中与转发去向
- S3：在 CI 内验证配置变更不破坏核心请求链路
- S4：团队内评审路由变更并安全回滚

## 5. 产品范围（Scope）

### 5.1 In Scope

- CLI 体验增强：`init`、`doctor`、`check-config`、`route explain`
- 本地联调增强：local override、请求回放
- 调试可观测：命中解释、链路日志、错误分类
- 测试友好能力：mock registry、CI 冒烟命令
- 协作能力：配置版本 diff、变更评审辅助
- 可扩展能力：插件 SDK（分阶段）

### 5.2 Out of Scope（当前 PRD 周期）

- 复杂 RBAC/多组织审批系统
- 多地域全自动流量调度系统
- 全功能可视化控制台（可做轻量版，不作为主线依赖）

## 6. 功能需求明细

## FR-1 初始化与运行检查

- 描述：提供初始化脚手架与环境自检能力
- 需求项：
  - FR-1.1 `service-router init` 生成模板配置与示例
  - FR-1.2 `service-router doctor` 检查端口、依赖、配置、连通性
  - FR-1.3 启动失败时输出可执行修复建议
- 验收标准：
  - 新用户在无先验知识下可在 10 分钟内成功代理请求

## FR-2 路由解释与调试能力

- 描述：提升“可解释性”和“可诊断性”
- 需求项：
  - FR-2.1 `route explain` 输出命中规则、匹配字段和优先级
  - FR-2.2 未命中时说明被哪些条件拦截（path/method/header）
  - FR-2.3 输出标准化 request-id 串联日志
- 验收标准：
  - 常见路由问题可在 5 分钟内完成初步定位

## FR-3 联调效率能力

- 描述：支持开发者对目标服务快速覆盖与回放
- 需求项：
  - FR-3.1 local override（按 path/header 重定向到本地服务）
  - FR-3.2 请求回放（基于日志/文件重放）
  - FR-3.3 联调模式下热更新反馈更及时
- 验收标准：
  - 联调准备时间相比基线下降 30%

## FR-4 测试与 CI 集成

- 描述：增强自动化验证能力
- 需求项：
  - FR-4.1 mock registry（内存注册中心模拟）
  - FR-4.2 `check-config` 静态校验配置
  - FR-4.3 `smoke-proxy` 执行最小链路验证
- 验收标准：
  - 关键服务配置变更可在 CI 冒烟阶段拦截高风险错误

## FR-5 协作与版本能力

- 描述：提升团队协作效率与变更安全性
- 需求项：
  - FR-5.1 配置版本对比（结构化 diff）
  - FR-5.2 变更说明模板（自动生成 review 摘要）
  - FR-5.3 快照/分享链接用于问题复现
- 验收标准：
  - 配置变更评审时长下降 30%

## FR-6 插件与生态能力（中长期）

- 描述：形成可扩展开发者生态
- 需求项：
  - FR-6.1 插件生命周期与扩展点定义
  - FR-6.2 官方插件示例（鉴权、头处理、路由策略）
  - FR-6.3 模板/插件分发机制（初版）
- 验收标准：
  - 30% 以上团队在一个季度内至少启用 1 个扩展能力

**本仓库工程切片**：`PluginMiddleware` async trait + `PluginChain` 有序执行链 + `server.plugins[]` YAML 配置已实现（Phase C）；内置 3 个官方插件（`request-logger`、`request-headers`、`response-headers`）对齐 FR-6.2；路由级 `response_headers` 保留为轻量扩展点。FR-6.3 插件分发（`dlopen` 外部加载）仍为后续目标，设计见 ADR 002。详见 **`docs/plugin-extension.md`**。

## 7. 非功能需求（NFR）

- NFR-1 性能：引入调试能力后，代理附加延迟控制在可接受阈值内
- NFR-2 稳定性：热更新和插件异常不影响基础代理主流程
- NFR-3 易用性：CLI 错误提示必须给出下一步可执行动作
- NFR-4 可观测：关键路径具备结构化日志和指标输出
- NFR-5 兼容性：升级时保留旧配置的向后兼容策略

## 8. 里程碑规划（与路线图对齐）

### M1（Y1 Q1-Q2）

- 交付：FR-1、FR-3（部分）、FR-4（基础）
- 目标：开箱即用 + 联调提效

### M2（Y1 Q3-Q4）

- 交付：FR-2、FR-4（完善）
- 目标：可诊断 + 可测试，达到开发者 GA

### M3（Y2 Q1-Q2）

- 交付：FR-5、FR-6（初版）
- 目标：可协作 + 可扩展

### M4（Y2 Q3-Q4）

- 交付：FR-6（增强）+ 多环境一致性能力
- 目标：生态化与规模化

## 9. 度量指标与目标值

- TTV（首次成功时间）：< 10 分钟
- DTR（平均排障时长）：下降 >= 40%
- SSR（自助解决率）：>= 70%
- CCR（配置变更一次成功率）：>= 95%
- CI 覆盖率：关键服务 >= 80% 接入 `check-config/smoke-proxy`

## 10. 风险与缓解

- R1：功能增多导致复杂度上升  
  - 缓解：命令分层（基础/高级），默认路径最短
- R2：调试日志影响性能  
  - 缓解：分级日志 + 可配置采样
- R3：插件生态导致兼容问题  
  - 缓解：稳定 API 合约与版本策略
- R4：团队资源不足导致里程碑延误  
  - 缓解：优先保障 FR-1/2/4，生态能力延后

## 11. 发布与运营建议

- 发布节奏：双月小版本，半年中版本，年度大版本
- 文档策略：每个新命令必须有示例、故障排查、FAQ
- 反馈机制：埋点 + issue 标签 + 用户访谈闭环
